1️⃣ WHAT IS ANNEXURE 5/A?
════════════════════════════════════════════════════════════════════════════

Annexure 5/A defines PASS TYPE ENROLLMENT LIMITS - a business rule engine
that controls how many of each type of person can be registered/enrolled under
a contractor firm.

Purpose:
  ✓ Prevent unauthorized person registrations
  ✓ Enforce safety & compliance rules
  ✓ Ensure proper supervision ratios
  ✓ Control gate pass generation limits

Real-World Analogy:
  Like a security checkpoint that says:
  - "Only 2 managers can register per company"
  - "Only 1 official representative per company"
  - "Must have 1 supervisor per 10 workers"
  - "Workers can be unlimited (based on work order)"


2️⃣ PASS TYPE RULES OVERVIEW
════════════════════════════════════════════════════════════════════════════

┌─────────────────────────────────────────────────────────────────────────┐
│ PASS TYPE 1: CONTRACTOR                                                 │
├─────────────────────────────────────────────────────────────────────────┤
│ Maximum Allowed: 2 per firm                                             │
│ Type: Fixed (Hard limit - cannot exceed)                                │
│ Override Allowed: Yes (Welfare Admin can override)                      │
│ Business Rule: Each firm can have maximum 2 contractors registered      │
│                                                                         │
│ Example: Contractor A already has 2 registrations                       │
│         Contractor A tries to add 3rd → BLOCKED ❌                      │
│         Welfare Admin can override → ALLOWED ✅ (with audit log)        │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ PASS TYPE 2: REPRESENTATIVE                                             │
├─────────────────────────────────────────────────────────────────────────┤
│ Maximum Allowed: 1 per firm                                             │
│ Type: Fixed (Strict - no flexibility)                                   │
│ Override Allowed: Yes (but generally not recommended)                   │
│ Business Rule: Only one official representative per firm                │
│                                                                         │
│ Example: Firm already has Representative "John Smith"                   │
│         Firm tries to add Representative "Jane Doe" → BLOCKED ❌        │
│         Cannot override without special permission                      │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ PASS TYPE 3: SUPERVISOR                                                 │
├─────────────────────────────────────────────────────────────────────────┤
│ Maximum Allowed: Dynamic (1 per 10 workmen + 1 additional)             │
│ Type: Ratio-based (Changes as workmen count changes)                    │
│ Override Allowed: Yes (Welfare Admin can add more)                      │
│ Business Rule: Must have proper supervision ratio                       │
│                                                                         │
│ Calculation Formula:                                                    │
│   allowed_supervisors = 1 + floor(total_workmen / 10)                 │
│                                                                         │
│ Examples:                                                               │
│   • 5-9 workmen    → 1 supervisor max                                   │
│   • 10-19 workmen  → 2 supervisors max                                  │
│   • 20-29 workmen  → 3 supervisors max                                  │
│   • 30-39 workmen  → 4 supervisors max                                  │
│   • 100 workmen    → 11 supervisors max                                 │
│                                                                         │
│ Why Dynamic? More workers need more supervision!                        │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ PASS TYPE 4: WORKMAN                                                    │
├─────────────────────────────────────────────────────────────────────────┤
│ Maximum Allowed: UNLIMITED (No fixed limit)                             │
│ Type: NoLimit (Depends on project work order)                           │
│ Override Allowed: N/A (No limit to override)                            │
│ Business Rule: Add as many workers as project requires                  │
│                                                                         │
│ Example: Company needs 500 workers for a project                        │
│         Can add all 500 without pass limit restrictions                 │
│         (Limited only by work order quantity)                           │
└─────────────────────────────────────────────────────────────────────────┘

4️⃣ DATA FLOW & MAPPING
════════════════════════════════════════════════════════════════════════════

DATA SOURCES FOR VALIDATION:

From Table: pass_limits (Default rules)
  ├─ Contractor → max_allowed = 2
  ├─ Representative → max_allowed = 1
  ├─ Supervisor → ratio_per_workmen = 10
  └─ Workman → max_allowed = NULL (unlimited)

From Table: contractors (Count contractors)
  └─ SELECT COUNT(*) WHERE firm_id = ? → current_count

From Table: annexure3a (Count representatives & supervisors)
  ├─ WHERE type = 'Representative' → current_representatives
  └─ WHERE type = 'Supervisor' → current_supervisors

From Table: workmen (Count total workmen)
  └─ SELECT COUNT(*) WHERE contractor_id = ? → workmen_count
     (Used for supervisor ratio calculation)

From Table: permanent_gate_passes (Historical passes)
  └─ For reporting & audit purposes

VALIDATION LOGIC FLOW:

Step 1: Get the type being added (Contractor | Representative | Supervisor | Workman)
         │
Step 2: Retrieve limit rule from pass_limits table
         │
Step 3: Determine rule type (Fixed vs Ratio vs NoLimit)
         │
Step 4a: If Fixed (Contractor, Representative)
         │  ├─ Query current count from related table
         │  ├─ Compare: current + requested <= max_allowed?
         │  └─ ALLOW or BLOCK
         │
Step 4b: If Ratio (Supervisor)
         │  ├─ Query workmen count from workmen table
         │  ├─ Calculate: allowed = 1 + floor(workmen / 10)
         │  ├─ Query current supervisors
         │  ├─ Compare: current + requested <= allowed?
         │  └─ ALLOW or BLOCK
         │
Step 4c: If NoLimit (Workman)
         │  └─ Always ALLOW (unless work order limit exceeded)
         │
Step 5: If BLOCK and override_flag = true
         └─ Check welfare admin permission
            ├─ If allowed → ALLOW with override
            └─ If not allowed → BLOCK

9️⃣ TROUBLESHOOTING
════════════════════════════════════════════════════════════════════════════

Problem: "Table 'pass_limits' not found"
Solution:
  1. Run: php api/annexure5a/init_pass_limits.php
  2. Check: mysqli_query($conn, "SELECT * FROM pass_limits")

Problem: "Supervisor calculation is wrong"
Solution:
  1. Verify workmen count is correct: 
     SELECT COUNT(*) FROM workmen WHERE contractor_id = ?
  2. Formula: 1 + floor(workmen / 10)
  3. Test: php api/annexure5a/EXAMPLES.php

Problem: "Override not working"
Solution:
  1. Check: override_allowed = true in pass_limits table
  2. Check: User has welfare admin role
  3. Test: validatePassLimit($conn, $id, $type, $count, true)

Problem: "Frontend validation not triggering"
Solution:
  1. Check: Script loaded - <script src="/js/annexure5a_validator.js"></script>
  2. Check: Browser console for JS errors
  3. Check: PassLimitValidator is defined in console

Problem: "Pass generation still works despite validation"
Solution:
  1. Check: api/generate_permanent_pass.php includes validator
  2. Check: validatePassLimit() is actually called (line 42-51)
  3. Verify: apiError() is thrown on validation failure
